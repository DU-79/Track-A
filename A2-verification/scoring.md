# A2：验证环境自动生成 - 评分细则

## 总分：100 分

10 个隐藏电路，每个电路 10 分。

## 总体结构

```text
总分 = sum(每电路得分)，共 10 个电路
每电路得分 = 骨架分 + 覆盖率分
```

| 项目 | 分值 |
|------|-----:|
| 验证骨架生成 | 3 |
| 行覆盖 + 分支覆盖 + 功能覆盖 | 7 |

**验证骨架是门禁项。** 若某电路的 testbench 无法编译、无法启动仿真、无法正确连接 DUT 或缺少有效的驱动/监视/比对器，该电路得 0 分，覆盖率不计。

## 1. 验证骨架（3 分）

骨架必须解析 RTL 接口并生成可运行的 testbench。必须支持至少 AXI/AXI-Lite、SRAM-like 和 valid/ready stream 接口。

| 子项 | 分值 | 要求 |
|------|-----:|------|
| RTL 接口解析 | 0.8 | 正确识别端口方向、宽度、时钟、复位、参数、接口分组 |
| 时钟与复位 | 0.6 | 稳定时钟、正确复位时序、复位后信号处于可用状态 |
| 输入驱动与输出监视 | 0.8 | 正确的驱动/监视握手、采样时机、反压处理 |
| 自动比对框架 | 0.8 | 参考模型或比对器可比对 DUT 结果并报告失败原因 |

**门禁失败条件（任意一条）：**

- Testbench 无法编译或仿真无法启动
- DUT 端口未连接、错连或宽度不匹配
- 时钟、复位或关键协议握手不可用
- 无有效驱动/监视；无法生成或采样事务
- 比对器未比对 DUT 行为或静默忽略比对失败

## 2. 覆盖率（7 分）

覆盖率从一次完整、可复现、通过的仿真运行中评测。

**综合覆盖率公式：**

```text
C = 0.4 * 行覆盖率 + 0.3 * 分支覆盖率 + 0.3 * 功能覆盖率
```

行覆盖率和分支覆盖率合计占综合分的 70%；功能覆盖率占 30%。三者均使用百分比（如 `96.90` 表示 `96.90%`）。

**功能覆盖率公式：**

```text
功能覆盖率 = (命中次数 > 0 的已覆盖 bin 数) / 有效 bin 总数 * 100%
```

一个 bin 被命中多次仅计为一个已覆盖 bin。功能 bin 必须对应可解释的功能场景、corner case、协议状态或场景组合。仅枚举随机值而无测试采样证据的 bin 不计。

| 综合覆盖率 C | 覆盖率得分 |
|-------------:|-----------:|
| C >= 85% | 7.0 |
| 65% <= C < 85% | 4.9 |
| 45% <= C < 65% | 2.8 |
| C < 45% | 0 |

**有效运行要求：**

- 固定 seed 和固定测试数以确保可复现
- `constraints.json` 描述随机变量、范围、协议约束和 seed
- `coverage_bins.json` 定义 coverpoint、bin 和对应功能场景
- `functional_coverage.json` 记录每 bin 命中次数、覆盖状态和汇总比例
- `coverage_result.json` 包含行覆盖率、分支覆盖率、功能覆盖率和综合覆盖率
- VCS/URG 报告必须对应当前提交的 RTL；cocotb 和纯 Verilog testbench 均可

**缺失或无效结果处理：**

- 仿真失败、挂起或 DUT 编译失败：覆盖率 = 0
- 缺少某一覆盖率类型：缺失项按 0% 计
- 功能 bin 无采样依据、结果不可解析或与定义不一致：功能覆盖率 = 0%
- 无可复现 seed 或输入序列：覆盖率上限 3 分
- 报告不对应当前 RTL：覆盖率 = 0

## 3. 示例：`case4`

`case4` 使用 cocotb 驱动 5000 个固定 seed 的 AXI 事务，VCS/URG 收集行和分支覆盖率，cocotb 采样功能 bin：

- 行覆盖率：`125 / 129 = 96.90%`
- 分支覆盖率：`25 / 31 = 80.65%`
- 功能覆盖率：`29 / 29 = 100.00%`

综合覆盖率：

```text
C = 0.4 * 96.90 + 0.3 * 80.65 + 0.3 * 100.00
  = 92.96%
```

`C >= 85%`，覆盖率 = 7 分。骨架门禁通过（3 分）：

```text
case4 得分 = 3 + 7 = 10 / 10
```

最终得分为所有 10 个电路得分之和。
